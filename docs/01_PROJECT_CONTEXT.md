# Project Context

## 1. Background

このプロジェクトは、AIソリューションエンジニアとしての活動と、コントラバス／エレキベース奏者としての活動を、別々に見せるのではなく、一つの人物像として伝えるために始まった。

当初の課題は、GitHubやnoteに多数の制作物がある一方で、一般企業や個人事業主から見たときに、何を相談できる人なのかが分かりにくいことだった。

不足しているのは実績そのものではない。実績が複数の場所へ分散し、技術者向けの言葉のままになっていることが主な課題である。

## 2. Existing Strengths

すでに次のような開発・改善経験がある。

- サーバー保守やネットワーク関連業務
- Excel上で作成していたラック図面を、ドラッグ操作で作れるWebアプリへ置き換える試作
- 複数人がExcelへ蓄積した虎の巻的な情報を横断検索する仕組み
- Threadsの自動投稿、予約、画像投稿、運用支援
- API、GAS、GitHub Actions、Python、Webアプリを組み合わせた自動化
- 音声生成、音楽、画像処理、AI漫画などのクリエイティブ制作

これらは「AIを使える」というより、現場の作業を観察し、動く仕組みへ落とし込める実績として扱う。

## 3. Primary Audience

主な対象は、地元の中小企業、個人事業主、店舗経営者。

対象者は必ずしもAIを導入したいわけではない。次のような具体的な困りごとを持っている。

- Excelや紙、PDFへの転記が多い
- 必要な情報を探すのに時間がかかる
- 同じ書類を何度も作っている
- SNS運用に時間を取られている
- 市販サービスが自社の業務に合わない
- AIで何ができるのか判断できない

## 4. Positioning

### Title

**AIソリューションエンジニア**

補足：

**業務改善・AI導入・自動化支援**

### Music Side

**コントラバス奏者／ベーシスト**

### Core Value

AIそのものを売るのではなく、現場の小さな面倒を聞き取り、使える仕組みに変える。

AIが適していればAIを使い、Excelやルール変更だけで十分なら、無理にAIを使わない。

## 5. Core Message

> 仕事の中にある「毎回面倒」を、AIと小さな仕組みで改善する。

補助メッセージ：

> AIを説明するだけでなく、現場で使える形までつくります。

## 6. Brand Concept

### SYSTEM / SOUND

システム設計と音楽には、次の共通点がある。

- 複雑な要素を整理する
- 全体の流れを設計する
- 不要なものを減らす
- 人が扱いやすい状態にする
- 最後に、心地よく動く状態をつくる

音楽家という側面は、技術者としての信頼を弱めるものではない。構造を捉える力、感覚、創造性、記憶に残る個性として活用する。

## 7. Relationship Style

大規模なコンサルティング会社のような距離感ではなく、約20人が入る小さな防音室で、演奏者と観客が近い距離にいるような関係を目指す。

- 相談しやすい
- 顔が見える
- その場の反応を見ながら調整できる
- 小さく試し、一緒に改善できる
- 専門用語を押し付けない

この「ライブ感」と「近さ」を、文章、写真、UI、問い合わせ導線に反映する。

## 8. Key Decisions from Discussion

- LP型ポートフォリオサイトと両面名刺を同じブランドで設計する
- サイトは作品一覧ではなく、相談につなげる営業用ポートフォリオとする
- ファイル形式を問わず何でも自動化できると表現しない
- 現状の作業、元データ、変更項目、運用方法を聞いてから手段を選ぶ
- 実績は技術名ではなく、課題・以前の作業・作ったもの・変化で説明する
- サイバーパンクや過剰なネオン表現を避ける
- F字孔のFとFrequencyのFをロゴ／シンボルの中心候補にする
- 黒、艶のある木、スポットライト、小さな赤をビジュアルの中心にする
- 大ホールではなく、小規模で親密なライブ空間を世界観とする
